Edit Eating Plan Profile

This use case details how the Dieter will use the system to edit their eating plan profile.

  1. Basic Flow
    1. Select Option

      2. The use case begins when the Dieter selects the option for editing a "My Eating Plan Profile".

    2. Display personalised eating plan

      3. The system displays the personalised eating plan.

    3. Edit Personal Details

      4. The Dieter edits his/her personal details [username, email, age, gender, height, weight]

    4. Choose Save Personalised eating plan

      5. The Dieter chooses to save the amended personalised eating plan.

    5. Calculate personalised eating plan

      6. The system re-calculates the personalised eating plan.

    6. Select Save eating plan

      7. The Dieter chooses to save the amended personalised eating plan.

    7. Save new personalised eating plan

      8. The system saves the amended personalised eating plan.

    8. Display amended eating plan

      9. The system creates and displays the Dieters's amended personalised eating plan, and the use case ends.

  2. Alternative Flows
    1. Enter medical condition

      2. If at step 1.3 (Edit personal details) the Dieter chooses to select one or more medical conditions. The system displays the appropriate screen. The user selects one or more medical conditions [allergies, diabetic, lactose intolerants, coeliac].

    2. Upload Photo

      3. If at step 1.3 (Edit personal details) the Dieter chooses to upload a photograph. The system displays the appropriate screen. The user chooses a photograph to upload, and selects the option to upload. The system uploads the photograph and adds it to the Dieterâ€™s Eating Plan Profile. The use case continues at step 1.3 (Edit personal details).

    3. Edit weight loss target

      4. If at step 1.3 (Edit personal details) the Dieter chooses to edit a weight loss target. The system displays the appropriate screen. The user edits the weight loss target. The use case continues at step 1.3 (Edit personal details).

    4. Edit dietary needs

      5. If at step 1.3 (Edit personal details) the Dieter chooses to edit dietary needs. The system displays the appropriate screen. The user edits the dietary need. [Vegetarian, vegan, kosher, low fat, low salt, low sugar, high in fibre, high in iron, contains vitamin c]. The use case continues at step 1.3 (Edit personal details).

    5. Cancel

      6. If at any time the Dieter chooses to cancel the activity, the system displays a warning message asking the Dieter to either continue with the cancel action or quit the cancel action. If the Dieter chooses to continue with the cancel action, the use case ends. If the Dieter chooses to quit the cancel action, the system closes the message and the use case continues from its current point.

    6. Database unavailable

      7. If at step 1.7 (Save new personalised eating plan) the system establishes that the database is unavailable, the system displays a warning message. The Dieter acknowledges the warning message and the use case ends.

  3. Key Scenario

    4. Aoife Murphy has a coeliac condition that she has managed since childhood. She needs to be able to cook for her family while maintaining a healthy weight. She wants to be able to cook tasty recipes that are gluten free and easy for her to source the ingredients.

  4. Preconditions

    5. The Dieter must have successfully logged in, and still be logged in.

  5. Post-conditions

    6. Either the personalised eating plan has been amended and all data has been saved, or no data has been modified.
